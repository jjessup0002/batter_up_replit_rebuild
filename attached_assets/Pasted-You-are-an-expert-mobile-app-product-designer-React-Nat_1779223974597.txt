You are an expert mobile app product designer, React Native/Expo developer, Flutter developer, UX architect, and youth baseball/softball coaching assistant.

I am building a mobile app called Batter Up.

The app is for youth baseball and softball coaches who need a simple, fast, offline way to manage batting lineups, track the current batter, track score, and optionally track detailed game stats.

The app must be easy enough for the average volunteer dad, mom, or little league coach to use during a live game without needing to understand complicated scorekeeping.

Build this as a polished mobile app, not just a prototype.

Recommended tech stack for Replit:
Because this project is being built in Replit, use React Native with Expo unless there is a clear reason Flutter is already configured and will work better in this Replit project.

The app must run on both Android and iOS.

Storage requirement:
The app must work fully offline.
Do not require a login.
Do not require a cloud database.
Do not require an internet connection during games.
Do not use Replit database, PostgreSQL, Supabase, Firebase, or any backend for app data.

All data must be stored locally on the user’s phone.

Recommended local storage:
Use AsyncStorage only for small settings.
Use SQLite, Expo SQLite, WatermelonDB, Realm, or another reliable on-device storage option for saved lineups, players, games, game events, and stats.

The app should be designed so cloud sync could be added later, but do not build cloud sync now.

Automatic next batter logic:
The app should automatically move to the next batter when an event completes the plate appearance.

Events that should move to the next batter:
- Hit
- Out
- Strikeout
- Walk
- Hit by pitch
- Reached on error
- Fielder’s choice
- Sacrifice
- Max pitches reached, if enabled
- Manual Next Batter

Events that should not automatically move to the next batter:
- Ball, unless it causes a walk
- Strike, unless it causes a strikeout
- Foul, unless league rules say it counts as final strike
- Run scored
- Score adjustment
- Stolen base

Ball and strike logic:
The app should support custom rules.

Default baseball rules:
- 4 balls = walk
- 3 strikes = strikeout

Supported custom rules:
- No walks
- No strikeouts
- 6 pitches max
- Swing until hit
- Foul on final pitch gives another pitch
- Foul on final strike counts as out
- Foul on final strike does not count as strikeout
- Coach manually chooses result when pitch limit is reached

Pitch limit logic:
For coach pitch leagues, support maximum pitches per batter.

Example:
Max pitches: 6

If the batter reaches the max pitch count, the app should apply the selected league rule:
- Batter is out
- Batter gets one more pitch if final pitch was foul
- Coach manually chooses result

Max runs per half inning:
The app should support max runs per half inning.

Example:
Max runs per half inning: 6

Inning logic:
The app should track:
- Inning number
- Top of inning
- Bottom of inning
- Current half inning
- Home team
- Away team
- Outs
- Runs this half inning

If the user’s team is away:
- The user bats in the top half of the inning

If the user’s team is home:
- The user bats in the bottom half of the inning

The app should move:
Top 1st → Bottom 1st → Top 2nd → Bottom 2nd

When the required number of outs is reached, or the max run limit is reached, the app should prompt or automatically move to the next half inning depending on settings.

At the start of each new half inning:
- Balls reset to 0
- Strikes reset to 0
- Outs reset to 0
- Runs this half inning reset to 0
- Pitch count resets to 0
- Current batter remains wherever the lineup rotation left off
- Batting order continues naturally across innings
